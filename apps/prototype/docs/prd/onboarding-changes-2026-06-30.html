The onboarding redesign — what to build

Where onboarding started yesterday vs. where it's landed today, written for the build: the capabilities we added, what we simplified, and the UI and data changes to make. Two surfaces changed — the Journey Builder and the Overview "tracker"; the client Doer is untouched.

🎯

What changed, in one line: two onboarding surfaces were redesigned — the agency Journey Builder became a guided, template-first wizard, and the agency Overview became a triage-first "who's stuck" tracker. The rule behind both: make the first run effortless without taking any capability away — power moved behind progressive disclosure, it wasn't removed.

This is the build list — the capabilities we added, what we simplified, and the UI + data changes to make. The product is three layers; only the Builder (§6) and the Overview (§8) changed, the client-facing Doer (§7) is untouched. Full spec: onboarding-and-activation-prd.md · whole-product tour: onboarding-and-activation-quickread.html.

The one principle behind both surfaces. Simplicity comes from smart defaults and progressive disclosure, not from removing power. A first-timer finishes on defaults alone; an expert can still open every control. Build for both on the same screen.

What to build — new capabilities

＋Capabilities we added

Behaviour that didn't exist before. Each is detailed in the matching deep-dive below.

A · Journey Builder

  • Template-first start — 3 ready, outcome-based templates (or start blank), so the owner never faces an empty canvas.
  • Guided two-pane wizard with a live client preview that updates as they build.
  • Per-step "Customize" — edit the title, the client-facing copy, the completion method, the how-to video, and split a step into per-asset sub-steps ("activate these 3 workflows" → 3 tracked tasks).
  • Locked auto-verify on standard steps — GoCSM always verifies real HighLevel state; the agency can't downgrade it. This is the wedge, now visible on every row.
  • Snapshot-driven asset picking — a step can require any asset or one specific named asset (workflow / funnel / form / pipeline) chosen from the agency's snapshot, asked once and remembered.
  • A calm "Live" summary for returning agencies, with Edit-jumps straight back into the wizard.

C · Overview tracker

  • A triage strip (Needs you / Waiting / On pace) that is the primary orientation and filters everything below.
  • A dollar-led verdict — one sentence naming the top bottleneck and the $/mo behind it (the one AI-marked surface).
  • A money-ranked "needs help" queue sorted by mrr × days stuck, with one owner-aware action per row.
  • An undoable action receipt — fire a fix and see scope + blast-radius + a ~5s undo before it commits.
  • Designed all-clear & "waiting on others" states — a real payoff when nothing needs you, and external waits kept out of the action queue.
What to build — simplifications

－What we simplified or removed

Friction taken out of the default path. None of these removed capability — the power moved behind a disclosure or a smart default.

A · Journey Builder

  • No blank 9-step canvas — replaced by a pre-filled template and one decision per page.
  • No blocking publish gate — operator jargon (A2P, webhooks) no longer stops publishing; it's publish-then-refine.
  • No drag-to-reorder — up/down arrows instead (safer for non-technical owners and accessibility).
  • Plain language everywhere — "carriers" → "phone companies", outcome-named steps, jargon out of the first run.
  • Advanced controls hidden, not gone — the deep editor (deep-links, plan variants, granular config) is one quiet "advanced editor" link away.

C · Overview tracker

  • Lead is triage, not a census — population counts are demoted into the full list, not the headline.
  • One AI surface only — just the verdict is AI; the queue ranking is plain arithmetic, not dressed up as AI. No ask-box, no generated prose.
  • One velocity number, never a chart — and a slip reads calm-neutral, not alarm-red.
  • No naked numbers — every figure carries a target, trend, or $; drill-downs stay collapsed, one open at a time.
  • "Pending" ≠ "stuck" — clients waiting on a carrier/DNS review sit in a quiet collapsed lane, off the action queue.

Action rules

  • Visible scope + blast radius before anything happens.
  • Grace countdown (~5s) with a live Undo.
  • Trigger Workflow = chosen, not auto-fired — picker of the client's real GHL workflows, smart-defaulted by stuck step.
  • One timing token ("Stuck N days") — a fact, not a scold; "Day N of 14" lives in the timeline drawer.

☑The build list — UI, then data

① In the UI

A · Journey Builder (§6)

  • Template-first Step 1 (3 templates + "start from blank").
  • 6-step two-pane wizard with persistent live preview.
  • Auto-verify locked for catalog steps; picker only for custom steps.
  • Per-step Customize: title · client copy · completion · video · sub-steps.
  • Asset steps: any/specific scope via remembered snapshot picker.
  • Order via arrows; publish-then-refine (no blocking jargon gate).
  • Returning user → JourneySummary; advanced = JourneyEditor.
  • Keep the auto-verify wedge visible on every row + the Step-2 header.

C · Overview (§8)

  • Triage strip leads; tiles filter the one client table.
  • Lifecycle census demoted to the full list (no 4th counter row).
  • Verdict dollar-led + "GoCSM Analysis" + "Numbers exact · wording is AI".
  • Agency verdict derives from blocked_by; queue ranked by mrr × days.
  • ActionReceipt with scope + grace + Undo on the primary action.
  • All-clear + sparse-aware copy; relief strip subordinate & collapsed.
  • One velocity number, calm-neutral on regression; no naked numbers.
  • StatusCard per-side borders; every verdict derives from the seed.
② In the data / the packet

The UI above sits on these. In the prototype most are front-end stubs — the build swaps each stub for the real source.

Builder
  • The step model gains client-facing instructions copy, a completion detector, an assetScope (any | specific) + a chosen snapshotAccountId and one named asset, and sub-steps.
  • Snapshot fetch — list a sub-account's workflows / funnels / forms / pipelines (today a snapshot.ts stub); the chosen sub-account is remembered across steps.
  • Auto-verify detection — standard catalog steps are verified by reading real HighLevel state (canAutoDetect); custom steps fall back to client-confirms / web-event / for-reference.
  • Brand colour auto-detected from the agency (a getHighLevelBrandColor stub today).
  • Templates are real catalog subsets, not placeholders.
Overview
  • blocked_by per stalled client (agency / client / external) drives the verdict, the whose-move pill, and the relief lane.
  • Queue ranking = mrr × days_on_current_step — arithmetic, computed, not AI.
  • Velocity = finished-this-week + average days + the delta vs the prior period.
  • The action receipt holds the fix for a ~5s grace window before it commits (undo), then reports back on completion; Trigger-Workflow lists the client's real GHL workflows.
  • Every verdict and number derives from live data — never hardcoded, so the headline, CTA and queue always agree.
🚧

Guardrails. Don't re-open the Doer (§7) — unchanged throughout. Keep one visible AI surface (don't label the ranking "AI"). Keep onboarding detection isolated from the Health-Score path (§4.1). Pending ≠ stuck — external-review clients stay out of the queue.
