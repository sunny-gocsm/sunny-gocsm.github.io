What we're building

Two surfaces, one engine

Everything operates on the Account (one HL sub-account) and a shared criteria/matching engine that answers “who matches right now / how many / who's about to.” Two surfaces sit on top.

🔔 Attention action layer

“What needs me right now?” A live queue of sub-accounts needing a play + two hero numbers (N need attention, $X MRR at risk), plus a “Step in” list for accounts a play ran on but didn't fix.

📘 Playbooks systems layer

A marketplace of 57 pre-built automations: browse → set up (3 steps) → run on the right accounts. Each play = a HighLevel workflow + an audience. This is the core product.

Epics E3–E6 · page /playbooks · embed /embed/playbooks

The most important cross-cutting rule

Phase 1 vs Phase 2 — the Health gate ★

Health is a gated, coined scoring system. Every surface must work in two phases, and the gate is enforced server-side (it's a trust/security rule, not cosmetic — NFR-4).

Phase 1 — Health NOT configured default / trial · the majority state

Fully functional on its own, using only HL-native data.

  • Available: logins, payment/billing status, workflow usage, plan, renewal date, feature adoption, raw NPS, MRR/spend, account.priority
  • Hidden: Health score, Health bands (Thriving/Healthy/Watch/At-Risk), lifecycle stages
  • Hidden: Health-derived queue signals, the rich match preview (band/breadth/forecast)

Phase 2 — Health configured additive, never a prerequisite

Unlocks additively — nothing that worked in Phase 1 changes or breaks.

  • Adds: Health score & bands, lifecycle stages, the “Health & Risk” filter group
  • Adds: Health-derived AttentionSignals, forecast7d, composition, band-aware previews
  • Still never exposed (either phase): PAS & raw pillar scores — never filterable, internal only
⚠️ Implementation rule: a single health-config.isConfigured() flag (per agency) drives every branch. In Phase 1, the serializer/selector layer must strip all health.* fields, bands, lifecycle, pillars from every response — catalog, accounts, queue, previews, restatement, copy. The marketplace “Situation” rating uses plain words that deliberately avoid the Health-band terms for exactly this reason.
Schema & business objects (§5 — canonical)

Domain model at a glance

All entities are agency-scoped (multi-tenant; account ids unique within an agency). “Live account” = enabled & not churned.

Account core
11 sub-objects: identity, ownership, status, lifecycle*, pipeline, health*, login, adoption, revenue, feedback, onboarding. (* = gated)
Signal
Event log. sticky & reverse & setup & ≤30d = account is losing a critical setup → the “save” plays.
Playbook ×57
title, problem, does, pre-written actions, category, effort, Situation rating, match(account)→bool.
Recipe ×6
A pre-seeded CriteriaSet + linked play; seeds the builder's empty state.
CriteriaSet
{match:"all"|"any", criteria[], nodes?} → Criterion (field·op·value) / Group (1 level only). The audience query.
AttentionSignal
A named (query + play + priority) that becomes a queue row. tier: native | health.
Attempt
A play ran but didn't fix — powers the “Step in” list. status flat/worse/unconfirmed · confidence high/low.
AutopilotStatus + OverseeMode
Per-agency deployment: off→on→paused→archived + governance auto/ease/review.
WorkflowDraft
Autosaved setup-in-progress, keyed by recipe; {step, criteria, workflowReady, savedAt}.
Criteria catalog (§5.2): 30 fields · 8 groups · ~7 “common”. Operators are plain words, never symbols (e.g. is more than, in the next, is any of). FieldTypes: score/money/number/days (numeric + between), dateRelative (inNext/inLast/moreThanAgo/within), enum (isAnyOf/isNoneOf/is/isNot), band P2, boolean, trendDir (falling/rising). An unknown field id ⇒ false (drop, never error).
Epic E6 · §5.7

Playbook lifecycle (state machine)

Per-agency, idempotent transitions. No hard deletes of a play that ever ran — soft archive only; restore always lands in paused (never directly live).

off — enable(oversee) → on ⇄ pause / resume ⇄ paused — archive → archived — restore → paused

Each transition has an exact confirmation toast (e.g. “Playbook paused”, “Archived — history kept”, “Restored — back in Paused”). disable (on|paused → off) exists but is not offered for plays with run history. Listed via GET /deployments?status=.

Epic E4 · the heart of activation

The unified 3-step setup & activation flow

One reusable flow opened from the marketplace, the Attention queue, or the Accounts table. Persistent header + stepper + footer. Autosaves a draft on every change (resume: “Picked up where you left off”). Publish is idempotent.

1
What it does
Video + the play's plain actions + an HL gate: “Open & modify it” → “I've completed this” sets workflowReady. Continue is locked until then.
2
When & who it runs on
The trigger builder (E5), seeded from a recipe (Phase-1 strips health.*). Live match count; Continue locked at 0 matches.
3
Review & publish
Two plain-English summaries + account preview (P2 rows / P1 count) + Autopilot toggle (default ON) → Publish.
Publish semantics: ongoing + autopilot ON ⇒ lifecycle off→on, runs continuously on new matches; autopilot OFF ⇒ one-time run over current matches; fixed-selection variant (opened with pre-picked accounts) runs ONCE on exactly those, no autopilot toggle. Double-publish is a no-op (idempotency key per agency·play·recipe / ·fixed-set). Draft cleared only on success.
Epic E5 · the engine + the UX

The trigger / audience builder — two modes

A live plain-English restatement sits at the top at all times (“Runs on accounts where…”) and the live count expands to show exactly who matches. The mode toggle is non-destructive.

Simple default · no AI

  • A prebuilt quick-add list grouped Account / Engagement / Billing / Users — tap to add a ready-made, editable condition.
  • An already-used field is hidden; “Browse all fields” opens the full categorized + searchable picker.
  • Flat conditions + one Match all | any switch.

Advanced AI on-ramp + rule builder

  • A natural-language “describe your audience” box → deterministic compile to editable rules (never auto-commits; “always check them” honesty gate).
  • A nested boolean builder: Match all/any · Add condition · Add group (one level only).
  • Switching back to Simple is disabled when genuine nesting exists — never silently flatten/reset.
💡 The live count is a disclosure: “N of your accounts match right now · See who” → Phase 1 shows a plain name + $/mo list (no Health); Phase 2 shows a richer preview (bands, a breadth meter, a 7-day forecast).

§9.A — why this wins (unoccupied white space)

The differentiators to protect in the build

1 · Live plain-English restatement. A generated sentence of the user's own rule — no competitor, not even HighLevel, shows this. Don't reduce it to a query summary.
2 · Live “who matches now” count + glanceable preview, updating on every edit, on a non-technical surface.
3 · NL → editable rules + live count. The NL drafts; the user lands in an editable builder (never a black box) with the count as ground truth.
4 · Point-of-use AI honesty (“always check them”), where competitors bury the disclaimer.
